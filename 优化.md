1. 检索阶段（不看 LLM 生成）
这是衡量「搜出来的东西对不对」，核心指标：

指标	含义	怎么算
Recall@k	相关文档被搜出来的比例	人工标 N 个问题，每个问题标注正确答案包含哪几条政策，看 top_k 里覆盖了多少
MRR	第一个相关结果排在第几位	对每个问题，取第一个命中相关政策的排名的倒数，求平均
NDCG@k	相关文档是否排在前面	综合考虑相关度和排名位置

2. 端到端（含 LLM 生成）
这是衡量「最终回答好不好」，维度更多：

维度	检查什么
来源准确性	回答引用的政策是否真实存在、标题/日期/机关是否正确
完整度	有没有遗漏关键相关信息
忠实度	回答内容有没有编造（幻觉）——RAG 最怕这个
相关性	回答是否聚焦用户问题，不跑题









用户提问 → LLM 决定调哪个工具 → 工具查 ChromaDB/DB → 格式化 → LLM 综合回答。

拆开来看就三步：

1. LLM 自主决策
收到问题后，LLM 从三个工具中选一个：

场景	调哪个	查什么
「新能源补贴有哪些」	    search_policies	            ChromaDB 语义搜索（语义检索，❌暂无bm25混合，✅已加Rerank精排+MMR多样性，🛡️三级自动降级；chunk的metadata直接继承
「第一条的全文发我」	    get_policy_detail	        SQLite 按 ID 取全文
「财政部2024年发了哪些文」	list_policies_by_metadata	SQLite 按条件精确筛选
2. 工具执行 + 格式化
以最常用的 search_policies 为例：


用户问: "新能源补贴" 
    → BGE-M3 向量化 (1024维) 
    → ChromaDB 余弦相似度搜 30 个 chunk 
    → 按 policy_id 分组合并 (同政策的 chunk 拼接，单篇上限 3000 字) 
    → 输出前 8 篇
3. LLM 综合回答
拿到格式化结果后，LLM 按系统提示词的要求（标注来源、禁止编造、按相关度排列）生成最终回答。

如果需要深入某篇，LLM 可以再调 get_policy_detail(policy_id=xxx) 拿全文，多轮工具调用直到觉得信息够了才输出最终结果。

# Reranker 降级链

为降低新人上手门槛，Reranker 支持三级自动降级：

```
cross-encoder (BGE-Reranker-v2-m3, ~2GB)
  ↓ 模型下载失败 / 推理报错
llm (DeepSeek V4 Flash, 零额外模型)
  ↓ API 调用失败
none (保留原始向量相似度顺序)
```

降级是自动且透明的：
- 降级发生时记录 WARNING 日志
- 单例会记住降级后的后端，本会话不再重试
- 用户可在 `.env` 中用 `RERANKER_BACKEND` 显式指定后端
- 国内用户设 `HF_ENDPOINT=https://hf-mirror.com` 可避免下载超时
