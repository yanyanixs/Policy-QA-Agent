# RAG 逻辑路线详解

以一次真实查询「**北京的数据要素政策**」为例，逐步拆解 RAG 管道的完整链路。

---

## 总览

```
用户输入 → LLM 决策（改写 Query + 选工具）→ Embedding 向量化
→ ChromaDB 语义检索 → Rerank + MMR 精排 → 去重格式化
→ LLM 综合生成 → 最终回答
```

本示例参数：

| 参数 | 值 |
|------|-----|
| LLM 模型 | `deepseek-v4-flash` |
| Embedding 模型 | `bge-m3`（Ollama 本地部署，向量维度 1024） |
| Reranker | `none`（未启用精排模型） |
| MMR | 启用，λ=0.7 |
| top_k | 8 |
| 总耗时 | 15153ms |

---

## Step 1：LLM 初次决策（1349ms）

### 输入

用户原始问题：`"北京的数据要素政策"`

### 系统提示词（核心约束）

LLM 被赋予 3 个可用工具：

| 工具 | 用途 |
|------|------|
| `search_policies` | 语义搜索政策知识库，支持按机关/地区/类型/日期过滤 |
| `get_policy_detail` | 按 ID 获取某条政策的完整正文 |
| `list_policies_by_metadata` | 按结构化字段精确浏览政策列表 |

### LLM 做了什么

1. **理解意图**：用户问的是"北京 + 数据要素"，属于开放性的政策搜索
2. **选择工具**：决定调用 `search_policies`
3. **改写 Query**：将 `"北京的数据要素政策"` 改写为 `"数据要素"`，并把 `"北京市"` 提取为过滤参数

```
原始 query:  "北京的数据要素政策"
     ↓ LLM 改写
改写 query:  "数据要素"
过滤条件:    location = "北京市"
    top_k:    8
```

### 为什么需要改写

用户自然语言通常包含冗余信息。LLM 将问题拆分为「语义核心」+「结构化过滤条件」：
- **语义核心** `"数据要素"` → 用于向量相似度搜索
- **过滤条件** `location=北京市` → 用于 ChromaDB 元数据过滤

这比直接用原始问题搜索更精准。

---

## Step 2：Embedding 向量化（4159ms）

### 输入

改写后的 query：`"数据要素"`

### 过程

调用 Ollama 本地部署的 **BGE-M3** 模型，将文本转为 1024 维向量：

```
"数据要素"
    ↓ BGE-M3 (Ollama)
[-0.0077, -0.0054, -0.0468, 0.0137, -0.0394,
 -0.0202,  0.0491,  0.0362, -0.0196,  0.0034, ...]
                                                    ← 共 1024 维
```

### 为什么用 BGE-M3

- 中文语义理解能力强
- 1024 维向量在精度和性能间取平衡
- 本地部署，无 API 调用延迟和费用

### 耗时分析

这一步耗时 4159ms，是单个步骤中最慢的之一。原因：
- BGE-M3 模型推理本身需要计算
- Ollama 是本地 HTTP 服务，有网络往返开销
- 这是每次查询都必须经过的环节，无法跳过

---

## Step 3：ChromaDB 向量相似度检索（317ms）

### 检索策略

```
检索数量: k = top_k × 3 = 8 × 3 = 24（上限 30）
过滤条件: location = "北京市"
```

采用 **过采样（oversampling）** 策略：需要 8 条结果，但检索 24 条。这为后续 Rerank + MMR 精排留出腾挪空间——如果只检索 8 条，精排阶段没有优化余地。

### 过程

1. 将 Step 2 的 1024 维向量送入 ChromaDB
2. ChromaDB 在向量空间中计算余弦相似度
3. 返回距离最近的 24 个 chunk

### 检索结果

```
原始返回: 24 个 chunk，来自 6 条不同政策

policy_id=9  (15 chunks) ← 最多，占 62.5%
policy_id=2  (2 chunks)
policy_id=53 (2 chunks)
policy_id=14 (1 chunk)
policy_id=5  (1 chunk)
policy_id=50 (2 chunks)
```

| 政策 | chunk 数 | 说明 |
|------|---------|------|
| 《北京市"数据要素×"实施方案（2024—2026年）》 | 15 | **核心政策**，chunk 0~16，内容覆盖面广 |
| 《医疗保障、气象服务领域"数据要素×"典型场景指引》 | 2 | 国家层面文件，部分命中 |
| 《关于征集第二批"数据要素×"典型案例的通知》 | 2 | 案例征集通知 |
| 《关于征集"数据要素×"典型案例的通知》(2024.02) | 1 | 第一次案例征集 |
| 《关于建设数据要素综合试验区的实施意见》 | 1 | 纲领性文件，但只命中 1 个 chunk |
| 《关于征集"数据要素×"典型案例的通知》(2025.04) | 2 | 第三次案例征集 |

### 观察

- policy_id=9 的 chunk 占比过高（62.5%），说明这篇文档与 query 高度相关，但也会导致结果**多样性不足**
- 这恰好是需要 Step 4（MMR）来修正的问题

---

## Step 4：Rerank + MMR 精排（0ms）

### 本示例情况

Reranker 后端 = `none`，即未启用精排模型。因此本步骤仅执行 **MMR 多样性重排**。

### MMR（Maximal Marginal Relevance）算法

MMR 的核心思想：在选择结果时，同时考虑**相关性**和**多样性**。

```
MMR 公式:
  MMR(chunk) = λ × 相关性分数 - (1-λ) × 与已选结果的最大相似度

λ = 0.7：偏向相关性（70%），但保留多样性惩罚（30%）
```

**贪心选择过程**：

```
初始候选: 24 个 chunk（按向量相似度排序）

第 1 轮: 选相似度最高的 chunk → policy_id=9 chunk=15
第 2 轮: 在剩余中，选 λ×相关 - (1-λ)×sim(已选) 最大的
         → 如果下一个也是 policy_id=9，相似度惩罚会让它得分降低
         → 如果出现 policy_id=2，它与已选的 policy_id=9 差异大，惩罚小
         → 最终 policy_id=2 chunk=1 得分更高，被选中
...
```

### 精排后结果

```
精排输出: 24 个 chunk，来自 6 条不同政策（与输入一致）

顺序变化:
  [0]  policy_id=9  chunk=15  ← 最相关，排第一
  [1]  policy_id=9  chunk=16
  [2]  policy_id=9  chunk=6
  [3]  policy_id=9  chunk=7
  [4]  policy_id=2  chunk=1   ← 非 policy_id=9 中最高，被 MMR 提前
  [5]  policy_id=9  chunk=13
  [6]  policy_id=53 chunk=0   ← 被 MMR 提前
  ...
```

### 如果启用 Reranker

| 后端 | 模型 | 效果 |
|------|------|------|
| `cross-encoder` | BGE-Reranker-v2-m3 (~2GB) | 用交叉编码器对 query-document 对逐一打分，精度最高 |
| `llm` | DeepSeek V4 Flash | 用 LLM 判断相关性，精度中等，无需额外模型 |
| `none` | 无 | 仅 MMR 多样性重排，纯向量相似度排序 |

Reranker 的作用是在粗筛后做**精细重打分**——ChromaDB 的向量相似度只是粗略的，Reranker 能更好理解 query 和文档的语义匹配程度。

## Step 5：去重与格式化（0ms）

### 去重

24 个 chunk 来自 6 条政策（policy_id 去重后 6 条）。同一政策的多个 chunk 会被合并。

```
policy_id=9:  15 chunks → 合并为 1 条政策记录
policy_id=2:   2 chunks → 合并为 1 条
policy_id=53:  2 chunks → 合并为 1 条
policy_id=14:   1 chunk  → 1 条
policy_id=5:    1 chunk  → 1 条
policy_id=50:   2 chunks → 合并为 1 条
─────────────────────────────────
最终: 6 条政策（≤ top_k=8）
```

### 格式化

将结构化的 metadata + chunk 文本拼接为 LLM 可读的格式：

```
--- 政策 1 ---
标题：北京市大数据工作推进小组关于印发《北京市"数据要素×"实施方案...》
发布机关：北京市政务服务和数据管理局
发布日期：2024-09-27
适用地区：北京市
政策工具：金融支持
发文字号：京大数据发〔2024〕1号
原文链接：https://zwfwj.beijing.gov.cn/...
政策ID：9
命中片段：16 处，全文 5561 字符
相关内容：
[政策正文...]
```

传给 LLM 的内容共 **7878 字符**。

## Step 6：LLM 生成最终回答（9328ms）

### 输入

消息序列：

```
[SystemMessage]  系统提示词（角色定义 + 工作准则）
[HumanMessage]   用户原始问题 "北京的数据要素政策"
[AIMessage]      LLM 的工具调用决策（search_policies）
[ToolMessage]    Step 5 的格式化结果（7878 字符的检索内容）
```

### LLM 输出

生成了 **1426 字符**的结构化回答：

1. **核心纲领性文件**：
   - 《"数据要素×"实施方案》—— 目标、领域
   - 《建设数据要素综合试验区的实施意见》—— 技术创新、服务体系、产业培育
2. **配套实施文件**：
   - 3 次典型案例征集通知的对比表格
3. **政策要点总结**：
   - 顶层设计先行、场景驱动落地、市场化配置改革

### 耗时分析

9328ms 占总耗时的 **61%**，是整个管道最慢的一步。原因：
- LLM 需要阅读 7878 字符的检索内容
- 需要综合多条政策信息，结构化输出
- DeepSeek API 的网络延迟 + 推理时间

## 关键设计决策

### 1. 为什么先 LLM 决策再检索

而不是直接用用户输入去检索？

- 用户输入包含口语化表达、指代词、冗余信息
- LLM 可以提取核心语义 + 结构化过滤条件
- 本例中 `"北京的数据要素政策"` → query=`"数据要素"` + location=`"北京市"`，比原文检索更精准

### 2. 为什么过采样（top_k × 3）

- ChromaDB 的向量相似度只是第一关粗筛
- 多检索 3 倍给 Rerank/MMR 留出优化空间
- 上限 30 防止极端情况（如 top_k=20 时检索 60 条）

### 3. 为什么需要 MMR

- 向量相似度检索会导致同一条政策的多个 chunk 霸榜（本例中 policy_id=9 占 62.5%）
- MMR 惩罚重复，保证用户看到不同政策，而非同一政策的 8 个片段
- λ=0.7 在「准」和「多样」之间取平衡

### 4. 为什么去重而不是直接截断

- 24 个 chunk 可能来自同一政策的多个片段
- 去重合并后按政策级别组织，LLM 更容易理解
- 避免了"回答了 8 条，其实只有 2 条政策"的情况

## 与 RAGAS 评估的衔接

运行 `trace_query.py` 后，输出的四要素直接对接 `RAGAS/evaluate.py`：

```bash
# 1. 追踪 + 输出四要素
cd backend
python scripts/trace_query.py "北京的数据要素政策" -g "真实答案..."

# 2. RAGAS 评估
cd ../RAGAS
python evaluate.py --from-trace ../backend/scripts/ragas_output.json -v
```

评估维度覆盖本文档描述的每个环节：
- **检索质量**（Step 3-4）：context_precision, context_recall, context_relevancy, context_entity_recall
- **生成质量**（Step 6）：answer_relevancy, answer_similarity, answer_correctness
- **事实一致性**（Step 6 vs Step 3-5）：faithfulness, hallucination_score
